# 멱등키 생성 방식 고민

> 날짜: 2026-05-06
> 원본 노션: [링크](https://www.notion.so/357edfd5a476800a812cc57b405e22b4)

---

### 멱등키를 db 저장 시(unique) 조회 or 생성 시 인덱스 단편화 발생

- 일반적으로 uuid v4 방식으로 저장 시 문자열로 인한 인덱스 단편화
해결방안

- TSID 또는 Snowflake ID
- UUIDv7






결론부터 말씀드리면, UUIDv4를 유니크 키로 사용하여 DB를 조회할 경우 정수형(BigInt)에 비해 성능 저하가 발생하는 것이 맞습니다. 특히 단순한 '조회' 속도보다 '삽입(Insert)'과 '인덱스 유지보수' 측면에서 치명적인 단점

---

### 1. UUIDv4 vs 정수형(BigInt) 성능 차이의 원인

UUIDv4가 정수형보다 RDBMS(특히 MySQL의 InnoDB 같은 B-Tree 기반 엔진)에서 성능이 떨어지는 이유는 크게 두 가지입니다.

- 인덱스 단편화 (B-Tree Page Split) - 🚨 가장 큰 문제
- 데이터 크기와 메모리 효율 (캐시 히트율 저하)
---

### 2. 효율적인 멱등키 생성 방식 추천

멱등키는 보통 클라이언트가 생성해서 서버로 보내는 경우가 많습니다. 상황에 따라 아래 세 가지 방식 중 하나를 선택하는 것이 가장 효율적입니다.

### 💡 추천 1: UUIDv7 (최근 가장 권장되는 방식)

UUIDv4의 "무작위성"으로 인한 DB 성능 저하 문제를 해결하기 위해 나온 새로운 표준입니다.

- 특징: 앞부분(Prefix)에 **생성 시간(Timestamp)**이 포함되어 있고, 뒷부분은 랜덤 값으로 채워집니다.
- 장점: 시간순으로 정렬이 가능(Time-ordered)하므로 DB에 삽입할 때 순차적으로 들어가 인덱스 단편화(Page Split)가 발생하지 않습니다. 정수형만큼은 아니지만 B-Tree 친화적이라 DB 성능 방어에 탁월합니다.
- 적용: 클라이언트(웹/앱)에서 UUID 라이브러리를 업데이트하여 UUIDv7을 생성해 보내도록 합니다.
### 💡 추천 2: TSID 또는 Snowflake ID (서버 중심 설계 시)

트위터에서 만든 Snowflake 패턴이나 TSID(Time-Sorted Unique Identifier)를 사용하는 방식입니다.

- 특징: 시간 정보 + 노드 ID + 시퀀스를 조합해 만든 64비트 정수형(BIGINT) ID입니다.
- 장점: 8바이트 정수형이므로 DB 조회 및 쓰기 성능이 완벽하게 최적화됩니다.
- 단점: 클라이언트 환경에서 완벽한 고유성을 보장하며 직접 생성하기 까다로울 수 있습니다. 보통은 서버가 멱등키를 미리 발급해 주고, 클라이언트가 그 키를 받아서 실제 요청 시 사용하는 구조에서 많이 씁니다.
### 💡 추천 3: 불가피하게 UUIDv4를 써야 한다면 (DB 최적화)

클라이언트 레거시 등의 이유로 무조건 UUIDv4를 받아야 한다면 DB 저장 방식을 바꿔야 합니다.

- VARCHAR(36) 대신 하이픈(-)을 제거하고 BINARY(16) 타입으로 변환하여 저장하세요.
- MySQL 기준 UNHEX(REPLACE(uuid, '-', ''))로 저장하고, 조회할 때 HEX()로 복원합니다. 공간을 36바이트에서 16바이트로 줄여 인덱스 크기를 최적화할 수 있습니다.
---

### 3. 아키텍처 관점의 꿀팁: RDBMS 대신 Redis 활용

멱등성은 영구적으로 보장해야 하는 데이터가 아닐 확률이 높습니다. (예: 결제 중복 요청 방지는 보통 최근 24시간~일주일 치만 방어하면 됨).

따라서 무거운 메인 RDBMS에 유니크 키를 걸어 매번 디스크를 뒤지게 하는 것보다, Redis 같은 인메모리 DB를 앞단에 두는 것이 훨씬 효율적입니다.

1. 클라이언트가 요청 시 헤더에 Idempotency-Key (UUID)를 담아 보냄.
1. 서버는 Redis에 SETNX (Set if Not eXists) 명령어로 해당 키를 저장 시도. (TTL 24시간 설정)
1. 성공하면 (처음 들어온 요청) 메인 로직 실행.
1. 실패하면 (이미 Redis에 키가 있음) 중복 요청으로 간주하고 기존 처리 결과를 반환하거나 차단.
요약하자면:

DB 성능을 최우선으로 고려하신다면 UUIDv7 도입을 가장 추천해 드리며, 트래픽이 많고 멱등성 체크 로직이 무겁다면 DB 앞단에 **Redis(SETNX)**를 배치하는 구조를 고민해 보시길 권장합니다.
