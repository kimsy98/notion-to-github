### RESTFUL

- URI는 자원(무엇을)을 가리키고
- HTTP 메서드는 행위(어떻게 할 것인가)를 지시
---

### 🌟 RESTful API 설계 5대 원칙

### 1. 자원은 동사가 아닌 명사로 표현(가장 중요)

URI는 '어떤 행동을 할 것인가'가 아니라 '어떤 데이터(자원)를 다룰 것인가'만 나타내야 합니다.

### 2. 명사는 복수형(Plural)을 사용한다

왜 단수(payment)가 아니라 복수(payments)를 쓸까요? DB의 테이블(레코드들의 집합)을 생각하시면 쉽습니다. "결제 내역(집합)들 중에서 특정 ID를 가진 하나를 콕 집어낸다"는 논리적 구조를 갖추기 위함입니다.

- GET /payment/1 (어색함: 단수에서 1번을 꺼낸다?)
- GET /payments/1 (자연스러움: 결제 내역'들' 중 1번을 꺼낸다)
### 3. 행위는 오직 HTTP 메서드로만 표현한다

CRUD(Create, Read, Update, Delete)를 URI에 절대 노출하지 않습니다.

- POST: 자원 생성 (Create)
- GET: 자원 조회 (Read)
- PUT: 자원 전체 수정 (Update)
- PATCH: 자원 일부 수정 (Update)
- DELETE: 자원 삭제 (Delete)
### 4. 자원의 계층(포함) 관계는 슬래시(/)로 표현한다

A라는 자원 안에 B라는 자원이 속해있을 때 사용합니다. (보통 2~3 Depth까지만 권장합니다. 너무 깊어지면 식별하기 힘듭니다.)

- 특정 유저의 지갑 조회: GET /users/{userId}/wallets
- 특정 결제 건의 승인 내역 생성: POST /payments/{paymentId}/approvals
### 5. 필터링, 정렬, 검색은 쿼리 파라미터(?)를 사용한다

새로운 URI를 만들지 않고, 쿼리 파라미터를 활용해 자원의 집합을 좁혀나갑니다.

---

### 💡 결론적으로 리뷰어의 피드백을 다시 보면:

질문자님이 작성하셨던 POST /ready는:

1. 명사가 아님 (준비하다/준비상태라는 동사적/형용사적 의미)
1. 복수형 집합이 아님 (어떤 자원을 다루는지 알 수 없음)
1. POST와 결합 시 의미가 어색함 (준비를 생성한다?)
그래서 자원 중심인 POST /payments (결제 건 집합에 새로운 결제 요청을 생성한다) 로 바꾸는 것이 REST의 정석입니다! 규칙이 딱 정리되시나요?
