### 해결 방안+확인해봐야 할 부분

1. 확정된 원인
jakarta.mail.AuthenticationFailedException: 535 5.7.1
Username and Password not accepted ... - nsmtp

- 535 5.7.1 = SMTP 표준 인증 실패 코드
- nsmtp = Naver SMTP 서버 시그니처
- 즉, Naver SMTP로 발송을 시도했으나 자격 증명이 거절됨
이 예외는 RuntimeException의 하위 예외(MailAuthenticationException)이며
CustomException이 아니므로,
GlobalExceptionHandler.handleException([GlobalExceptionHandler.java:88](http://globalexceptionhandler.java:88/)-95
(api/src/main/java/co/sft/sns/presentation/controller/GlobalExceptionHandler.j
ava#L88))의 fallback 핸들러를 타고 HTTP 500 + "Internal server error"로
응답됐습니다.

★ Insight ─────────────────────────────────────

- 트레이스에 userProfileRepository.findByEmail SELECT 쿼리가 함께 찍혔다는 점
= 요청은 purpose=PASSWORD_RESET 케이스였고, 사용자 조회까지는 성공한 뒤 발송
단계에서 실패했다는 뜻입니다. 따라서 DB/Redis는 정상, 코드 로직도 정상이며,
순수하게 SMTP 인증 자격 문제입니다.
- GlobalExceptionHandler는 모든 RuntimeException을 500으로 매핑하기 때문에,
응답만으로는 원인을 식별할 수 없습니다. 로그의 stack trace가 유일한 진단
경로였고, 이번에 확보된 것이 그 trace입니다.
- 핵심 교훈: 외부 시스템 호출(SMTP/Redis/HTTP)을 하는 인프라 레이어는 발생
가능한 예외를 도메인 예외로 래핑해두는 것이 좋습니다. 그래야 글로벌 핸들러가
의미 있는 HTTP 코드(예: 502/503)를 줄 수 있고, 사용자 응답에 원인 단서가
남습니다.

1. 지금 본인이 확인해야 할 것 (우선순위 순)
✅ A. dev 컨테이너의 메일 환경변수 값 확인 (필수)

# dev 컨테이너에 접속해서

env | grep -i MAIL_

기대값 (Naver Mail 기준):
MAIL_HOST=[smtp.naver.com](http://smtp.naver.com/)
MAIL_PORT=587
MAIL_USERNAME=실제계정@naver.com   ← 풀 이메일이어야 함. 도메인이나 ID만 X
MAIL_PASSWORD=앱비밀번호            ← 일반 로그인 비번 X
MAIL_PROTOCOL=smtp

가장 흔한 실수: MAIL_USERNAME에 [timora.co.kr](http://timora.co.kr/) 같은 도메인이나 myaccount 같은 ID
부분만 넣음. → Naver SMTP는 @ 포함 풀 이메일을 요구함.

✅ B. Naver Mail SMTP 활성화 여부 확인 (필수)

Naver Mail에 로그인 → 환경설정 → POP3/IMAP 설정 → POP3/SMTP 사용함 체크되어
있는지 확인.

기본 상태에서는 SMTP가 꺼져 있어, 비밀번호가 맞아도 535로 거절됩니다.

✅ C. 2단계 인증 사용 중이면 앱 비밀번호 발급 (조건부)

Naver Mail 계정에 2단계 인증을 켜둔 상태라면, 일반 비밀번호로는 SMTP 인증이
되지 않습니다.

내정보·보안 → 2단계 인증 → 애플리케이션 비밀번호 관리에서 SMTP용 앱 비밀번호
생성 → 그 값을 MAIL_PASSWORD에 넣기.

✅ D. Naver Works 사용이라면 호스트/계정 확인

회사 도메인 메일(Naver Works)이면 호스트가 다릅니다:
MAIL_HOST=[smtp.worksmobile.com](http://smtp.worksmobile.com/)
MAIL_USERNAME=계정@회사도메인.co.kr
SMTP 활성화도 Works 관리자 콘솔에서 별도 설정해야 함.

✅ E. 잠금 상태 확인 (반복 실패 시)

위 A~D가 모두 맞아도 안 되면, 반복적인 인증 실패로 Naver가 일시적으로 잠갔을
가능성이 있습니다.

- 해당 계정으로 웹 로그인 시 캡차/잠금 안내가 뜨는지 확인
- 10~30분 대기 후 재시도
⚠️ F. (보조) 트래픽이 SMTP 서버에 실제로 닿는지 확인

위까지 해도 안 되면 dev 컨테이너에서:
nc -zv [smtp.naver.com](http://smtp.naver.com/) 587

# 또는

openssl s_client -starttls smtp -connect [smtp.naver.com:587](http://smtp.naver.com:587/)
→ 핸드셰이크가 거기까지 가는지로 네트워크 단절 vs 자격 거절을 구분.
