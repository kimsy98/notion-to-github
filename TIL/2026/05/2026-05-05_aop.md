# aop

제공해주신 코드는 AOP(관점 지향 프로그래밍)의 강력함을 아주 잘 보여주는 훌륭한 실무 예제입니다. AOP는 이전에 학습하셨던 1번: Spring Core (핵심 원리) 🏗️ 파트에서 의존성 주입(DI)과 함께 가장 중요한 두 축 중 하나를 담당하고 있죠.

이 코드는 결제 시스템에서 필수적인 '멱등성(Idempotency - 동일한 요청을 여러 번 보내도 결과가 같아야 함)'을 보장하기 위해, 핵심 비즈니스 로직(결제 처리) 앞뒤로 중복 요청 검사 로직을 분리해 낸 것입니다.

AOP의 핵심 개념(Aspect, Pointcut, Advice, JoinPoint)을 중심으로 이 코드를 한 줄씩 깊게 파헤쳐 보겠습니다.

---

## 1. AOP 클래스 선언부: 관점(Aspect)의 정의

```java
@Aspect
@Component
public class IdempotencyAspect {
```

- @Aspect: 이 클래스가 단순한 객체가 아니라, 관점(Aspect)을 정의한 클래스임을 스프링에게 알립니다. Aspect는 '어디서(Pointcut)' + '무엇을 할지(Advice)'를 하나로 묶은 모듈입니다.
- @Component: 스프링 AOP는 스프링 빈(Bean)으로 등록된 객체에 대해서만 동작합니다. 따라서 이 Aspect 자체도 스프링의 관리 하에 두기 위해 빈으로 등록합니다.
## 2. 어드바이스(Advice)와 포인트컷(Pointcut): 언제, 어디에 적용할 것인가?

```java
@Around("@annotation(com.practice.paymentassignment.Idempotent)")
```

- @Around (Advice): AOP에서 가장 강력한 Advice 타입입니다. 대상 메서드의 실행 전(Before)과 후(After), 심지어 예외가 발생했을 때까지 모두 제어할 수 있으며, 대상 메서드의 실행 자체를 막거나 리턴 값을 조작할 수도 있습니다.
- "@annotation(...)" (Pointcut): 이 부가 기능(멱등성 체크)을 어디에 적용할지 타겟을 지정하는 표현식입니다. 여기서는 @Idempotent라는 커스텀 어노테이션이 붙은 메서드들만 콕 집어서 이 로직을 적용하겠다는 의미입니다. 핵심 로직 코드에 @Idempotent 하나만 붙이면 이 복잡한 코드가 알아서 작동하게 됩니다.
## 3. 조인포인트(JoinPoint): 가로챈 대상의 정보

```java
public Object checkIdempotency(ProceedingJoinPoint joinPoint) throws Throwable {
```

- ProceedingJoinPoint: AOP가 가로챈 '원래 실행되어야 할 대상 메서드'에 대한 모든 정보를 담고 있는 객체입니다. @Around Advice에서만 사용할 수 있는 특별한 객체로, 내부적으로 proceed()라는 메서드를 가져서 개발자가 원하는 타이밍에 원본 메서드를 실행시킬 수 있는 권한을 줍니다.
- 반환 타입이 Object인 이유는 가로챈 대상 메서드가 무엇을 반환할지 알 수 없기 때문입니다. 타겟 메서드가 실행된 후 그 결과값을 그대로 다시 돌려주기 위함입니다.
## 4. 사전 처리 (Pre-processing): 원본 메서드 실행 전 개입

```java
ServletRequestAttributes attributes = (ServletRequestAttributes) RequestContextHolder.getRequestAttributes();
// ... (생략: HTTP 요청에서 Header 값 추출) ...
MethodSignature signature = (MethodSignature) joinPoint.getSignature();
Method method = signature.getMethod();
Idempotent idempotent = method.getAnnotation(Idempotent.class);
```

- 이 부분은 타겟 메서드가 실행되기 전에 가로채서 수행하는 로직입니다.
- joinPoint.getSignature(): 가로챈 원본 메서드의 시그니처(이름, 파라미터 타입, 리턴 타입 등) 정보를 가져옵니다. 이를 통해 원본 메서드에 붙어있는 @Idempotent 어노테이션의 설정값(prefix, ttl 등)을 동적으로 읽어옵니다. AOP 덕분에 하드코딩 없이 유연한 프로그래밍이 가능해집니다.
```java
Boolean isFirstRequest = redisTemplate.opsForValue().setIfAbsent(redisKey, "PROGRESSING", idempotent.ttl(), TimeUnit.SECONDS);

if (Boolean.FALSE.equals(isFirstRequest)) {
    log.warn("중복 요청 감지 - 멱등성 키: {}", redisKey);
    throw new AlreadyProcessedException("이미 처리 중이거나 완료된 주문입니다.");
}
```

- AOP의 강력함이 드러나는 구간입니다. 레디스를 확인해서 이미 처리 중인 요청이라면, joinPoint.proceed()를 호출하지 않고 곧바로 예외를 던져버립니다.
- 즉, 중복 결제 요청이 들어오면 핵심 비즈니스 로직(결제 DB 접근, 외부 API 연동 등)은 아예 실행조차 되지 않고 방어됩니다. 비즈니스 로직은 "이 요청이 중복인가?"를 전혀 신경 쓸 필요 없이 본연의 역할(결제)에만 집중할 수 있게 됩니다.
## 5. 핵심 로직 실행 (Target Execution)

```java
try {
    return joinPoint.proceed();
```

- joinPoint.proceed(): AOP 로직의 꽃입니다. "사전 검사가 끝났으니, 이제 원래 하려던 타겟 메서드(핵심 비즈니스 로직)를 실행해라"라는 명령입니다.
- 이 메서드가 호출되면 제어권이 타겟 메서드로 넘어갔다가, 타겟 메서드의 실행이 끝나면 다시 이 AOP 클래스로 결과물(또는 예외)과 함께 제어권이 돌아옵니다.
## 6. 사후 처리 및 예외 제어 (Post-processing & Exception Handling)

```java
} catch (Exception e){
    if (e instanceof BusinessException) {
        throw e;
    }
    // 5xx (서버 에러, DB 다운 등) -> 락 삭제 (캐싱 X, 재시도 허용)
    redisTemplate.delete(redisKey);
    throw e;
}
```

- proceed() 실행 중(즉, 비즈니스 로직 처리 중) 발생한 예외를 AOP에서 다시 가로채어 후속 처리를 합니다.
- 사용자의 잘못된 요청(BusinessException)이면 락을 유지하지만, 서버 측의 일시적인 에러(DB 다운 등)라면 사용자가 나중에 다시 시도할 수 있도록 Redis에서 멱등성 키(락)를 지워주는 세밀한 제어를 하고 있습니다.

## 💡 요약하자면

이 클래스는 AOP를 활용해 "어떤 컨트롤러나 서비스 로직이든 @Idempotent만 붙어있다면, 그 메서드가 실행되기 전에 내가 먼저 HTTP 헤더를 확인하고 Redis에 락을 걸겠다. 중복이면 튕겨내고, 아니면 통과시켜 주며, 만약 서버 에러가 나면 내가 알아서 락을 풀어주겠다" 라는 문지기(Proxy) 역할을 완벽하게 수행하고 있습니다.
