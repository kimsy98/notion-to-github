### 1. 용어 교정: 'DB 클러스터링' vs '분산 스케줄링(Clustering)'

작성하신 내용 중 *'DB 클러스터링이란 하나의 데이터베이스를 여러 서버가 나누어서 구축하는 것'*이라고 하신 부분은 데이터베이스 서버 자체를 여러 대 두는 기술(Replication 등)에 가깝습니다.

Quartz에서 말하는 기능은 'DB를 활용한 분산 스케줄링(Clustered Scheduling)'입니다.

여러 대의 애플리케이션 서버(Spring 서버)가 하나의 공통 DB를 바라보면서 스케줄 작업을 조율하는 것입니다. A 서버가 스케줄을 실행하려고 DB에 락(Lock)을 걸면, B 서버는 DB 상태를 보고 "아, A가 이미 하고 있네" 하고 작업을 건너뛰어 중복 실행(동시성 문제)을 방어하는 원리입니다.

### 3. 실무 도입 가이드: 언제 무엇을 써야 할까?

🟢 Spring Scheduler를 써야 할 때

- 현재 운영 중인 서버가 딱 1대일 때.
- 서버가 여러 대라도, 각 서버마다 독립적으로 실행되어야 하는 작업일 때 (예: 각 서버의 로컬 메모리 캐시 초기화, 개별 서버 헬스체크 등).
- 단순하고 가벼운 백그라운드 작업이 필요할 때.
🟠 Quartz를 써야 할 때

- 서버가 2대 이상(다중 서버 환경)이고, 전체 시스템을 통틀어 무조건 딱 1번만 실행되어야 할 때 (예: 매일 자정 정산 처리, 사용자 단체 이메일/푸시 발송 등).
- 작업이 실패했을 때 반드시 재시도하거나 복구해야 하는 핵심 비즈니스 로직일 때.
- 관리자 페이지에서 스케줄 시간을 수시로 껐다 켰다 하거나 변경해야 할 때.
💡 최신 실무 팁 (대안 기술)

"다중 서버에서 중복 실행만 막고 싶은데, Quartz는 너무 무겁고 설정이 복잡하다"라고 느끼는 경우가 많습니다.

이럴 때는 가벼운 대안으로 ShedLock이라는 라이브러리를 Spring Scheduler와 결합하여 사용하거나, DB 비관적 락 혹은 Redis의 원자적 연산(SETNX 등)을 활용한 분산 락을 직접 구현하여 스케줄러 메서드에 적용하는 방식도 실무에서 아주 많이 사용됩니다. 복잡한 시스템의 결합도를 낮추고 동시성만 깔끔하게 제어하기 좋은 접근입니다

1. 스케줄러 도입
@Scheduled(fixedRate = 60000)

public void updateExpiredPayments() {

List<Payment> expiredPayments = paymentRepository
.findAllByStatusAndExpiredAtBefore(PaymentRequestStatus.READY, Instant.now());
expiredPayments.forEach(Payment::markAsExpired);
}

장점

- 데이터 일관성: 주기적으로 DB를 직접 업데이트하므로 조회 시 항상 최신 상태 보장
- 분산 환경友好: 여러 서버 인스턴스가 있어도 스케줄러는 하나만 동작하게 설정 가능
- 조회 성능: 추가 계산 없이 바로 status 필드만 읽으면 되므로 조회 쿼리가 빠름
- 예측 가능: 1분마다 실행 등 정해진 로직이므로 동작 예측이 쉬움
단점

- DB 부하: 1분마다 SELECT + UPDATE가 발생 (READY 상태가 많을 때 치명적)
- 지연 발생: 최대 1분까지 만료 상태 반영 지연 (사용자가 만료된 결제를 READY로 볼 수 있는 시간)
- 추가 복잡도: 스케줄러 중복 실행 방지를 위한 분산 락(ShedLock 등) 필요
- 인프라 비용: 별도 스케줄링 인프스 관리 필요
- 오버헤드: 만료될 결제가 없을 때도 불필요한 쿼리 실행
적합한 경우

- 결제 요청이 많지 않을 때 (일일 수천 건 이하)
- 조회 빈도가 높고 조회 성능이 중요할 때

1. 조회 시점에 계산만 (Getter 메서드)
public PaymentRequestStatus getEffectiveStatus() {
if (status == PaymentRequestStatus.READY && isExpired()) {
return PaymentRequestStatus.EXPIRED;
}
return status;
}

장점

- 구현 단순: 코드 한 줄로 해결
- DB 부하 없음: 조회 시 메모리에서만 계산
- 즉시 반영: 만료되자마자 조회하면 바로 EXPIRED로 보임
단점

- 불일치: DB에는 여전히 READY, 조회 결과만 EXPIRED → 데이터 불일치로 버그 유발 가능
- 모든 조회 로직 변경 필요: getStatus() 대신 getEffectiveStatus()를 호출하도록 전체 코드 수정
- 인덱스 활용 불가: status로 필터링하는 쿼리에서 만료된 결제가 걸러지지 않음
- 혼란 가능성: 개발자가 getEffectiveStatus() 호출을 빼먹으면 버그
적합한 경우

- 거의 사용하지 않음, 데이터 불일치가 치명적인 프로젝트에서는 부적합

1. JPA @PostLoad 훅
@PostLoad
public void postLoad() {
if (status == PaymentRequestStatus.READY && isExpired()) {
this.status = PaymentRequestStatus.EXPIRED;
}
}

장점

- 자동화: 개발자가 신경 안 써도 됨, 조회 시 자동으로 상태 변경
- 코드 분리: 비즈니스 로직에 상태 체크 코드 섞이지 않음
단점

- 영속성 컨텍스트에서만 변경: Transaction이 끝나면 변경 사항이 DB에 반영되지 않음
- 일회성: 다시 조회하면 다시 READY로 로딩 → 다시 EXPIRED로 변경 → 무한 반복
- 디버깅 어려움: 언제 상태가 바뀌는지 추적이 어려움
- JPA 의존: 구현 변경 시 영향
적합한 경우

- 거의 사용하지 않음, @PostLoad의 특성상 이 용도에는 부적합
